# Reference-voice bank

The Zonos engine conditions synthesis on a speaker embedding derived from a short reference clip.
This directory holds one clip per reference-voice id the plugin's `ZonosVoiceMap` can request, so
the engine can resolve any plugin `voice{race,gender,player}` to a concrete embedding.

## Required clips

One `<id>.wav` is required for every id below (the build script `packaging/build_bundle.py` asserts
this before producing a bundle and fails the build if any is missing):

- `narrator_neutral` (default / fallback voice)
- `player_male`, `player_female`
- `human_male`, `human_female`
- `elf_male`, `elf_female`
- `dwarf_male`, `dwarf_female`
- `goblin_male`, `goblin_female`
- `troll_male`, `troll_female`
- `undead_male`, `undead_female`
- `demon_male`, `demon_female`
- `wizard_male`, `wizard_female`

Each clip should be a few seconds of clean mono speech with the character of that race/gender.

## How the clips are produced

The clips are **generated from the Kokoro CPU engine**, not hand-recorded or externally sourced.
`scripts/generate_reference_voices.py` drives the Kokoro engine's `--stdio` protocol once per voice
id, on a fixed neutral reference phrase, using the Kokoro speaker for the **same** race/gender/player
(via the engine's `SpeakerMatrix` / `VoiceProfile` matrix). So `dwarf_male.wav` is cloned from
Kokoro's dwarf-male voice, `narrator_neutral.wav` from Kokoro's default speaker, and so on. This
keeps voice identity consistent between the CPU (Kokoro) and GPU (Zonos) backends and keeps the whole
pipeline in-house and license-clean. Each clip is a few seconds of clean **mono 16-bit PCM at
24000 Hz**, exactly what Zonos wants for reference conditioning.

Run it locally after building the Kokoro engine image (see `docs/engine-pipeline.md`):

    ./gradlew :engine:engineImage   # + stage the kokoro-multi-lang-v1_0 model into the image's model/
    python engine-zonos/scripts/generate_reference_voices.py \
        --engine-launcher engine/build/engine-image/kokoro-engine

The Zonos release workflow (`.github/workflows/zonos-engine-release.yml`) runs the same generator
before `build_bundle.py`, so a dispatched release produces a complete bank with no manual staging.

## Status

The `.wav` clips are **not committed to the repo** (they are generated audio assets, kept out of the
Java plugin jar and out of source control via `.gitignore`). The generator (locally or in the release
workflow) produces them on demand into this directory before `build_bundle.py` runs.

## Licensing

All clips are **generated by the Kokoro TTS model** (`kokoro-multi-lang-v1_0`) via sherpa-onnx. Both
Kokoro and sherpa-onnx are **Apache-2.0** licensed, so the generated clips are license-clean and used
in-house. They are used only to derive a Zonos speaker embedding at runtime and are not redistributed
as a standalone audio product.
